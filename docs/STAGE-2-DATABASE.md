# Stage 2 — Database Models and Initial Migration

## Scope

Stage 2 implements the database layer fixed in Stage 1. It contains 64 PostgreSQL tables, SQLAlchemy 2.x ORM mappings, an explicit Alembic initial revision, named constraints/indexes, integrity triggers, and schema-contract tests. It intentionally does not implement Docker, runtime configuration, bot handlers, queues, workers, API routes, or business services from later stages.

## Delivered schema domains

- Identity, consent, plans, subscriptions, quota buckets/reservations/ledger, bans, restrictions, strikes, and appeals.
- Durable jobs, attempts, events, transactional outbox, application instances, worker leases, Telegram capability registry, and webhook deduplication.
- Files, ownership references, scan results, download links/sessions, distinct stream tokens/sessions, media variants/segments, Nginx bandwidth usage, and storage snapshots.
- Forced membership, advertisements, public-channel workflow/reports, support tickets/messages, and resumable broadcasts.
- Administrator roles/permissions/overrides/scopes/sessions, confirmations, independent approvals, append-only audit, settings/history/profiles, and translation overrides.
- Security events/actions, domain/hash blocklists, backup destinations, backups, and restore history.

## Database-level invariants

- UUID primary keys are generated by the application; Telegram IDs and byte counters use `BIGINT`; timestamps are timezone-aware UTC values.
- PostgreSQL check constraints reject negative counters, invalid periods, invalid status domains, short token hashes, impossible storage totals, and quota buckets whose committed plus reserved amount exceeds their limit.
- Partial unique indexes enforce one active subscription per user, one active reservation per job and quota bucket, one pending appeal per restriction, and one active public post per request/channel.
- A job-state trigger rejects transitions outside the Stage 1 state machine.
- Usage ledger rows, job events, admin audit rows, and settings history are append-only. Corrections must be new compensating records.
- Last-active-Super-Admin removal is serialized with a PostgreSQL transaction advisory lock and rejected when it would leave no active Super Admin.
- System administrator-role identity is immutable, and the requester cannot satisfy an independent multi-admin approval.
- Every mutable timestamped table has a database `updated_at` trigger, so bulk SQL updates cannot silently leave stale timestamps.
- Tokens/secrets are represented only by hashes or external secret references; no raw download, stream, admin-session, or worker-lease token column exists.

## Migration behavior

The frozen revision is `0001_initial_schema`. It contains explicit PostgreSQL DDL statements and does not call `Base.metadata.create_all()` or `drop_all()`. Upgrade runs transactionally. Downgrade explicitly drops all 64 tables and integrity functions in dependency-safe/CASCADE order and is intended only for a disposable database or an explicitly authorized rollback.

The Alembic environment accepts `DATABASE_URL`. The value in `alembic.ini` is a non-secret development-only example; production credentials must be supplied by the future Docker-secret/runtime configuration layer.

Commands after installing the pinned Stage 2 dependencies:

```bash
DATABASE_URL='postgresql+psycopg://...' alembic upgrade head
DATABASE_URL='postgresql+psycopg://...' alembic current
```

Offline SQL review:

```bash
DATABASE_URL='postgresql+psycopg://user:password@localhost/db' alembic upgrade head --sql
```

## Test safety

Static metadata/migration tests run without a database. The PostgreSQL migration test is deliberately skipped unless both conditions hold:

1. `TEST_DATABASE_URL` points to a database whose name starts with `mdlbot_test`.
2. `MDLBOT_ALLOW_DESTRUCTIVE_SCHEMA_TESTS=1` is set explicitly.

The integration fixture refuses a non-empty database and uses Alembic downgrade instead of `Base.metadata.drop_all()`. This avoids accidentally destroying a developer or production schema.

## Transaction boundary contract for later services

Quota, job, outbox, file-finalization, and accounting services must use caller-owned SQLAlchemy transactions. Repositories may flush but must not call `commit()` internally. The admission transaction will create/update all applicable quota buckets in a stable lock order, create per-window reservations, create the durable job, and enqueue an outbox row before committing once.
